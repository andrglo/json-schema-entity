# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A library that maps a tree of parent/child SQL tables (joined by foreign keys) into a single
nestable JSON document — an "entity" — so a relational schema can be read and written like a
NoSQL document. Supports PostgreSQL and SQL Server. Published to npm as `json-schema-entity`;
`main` is `src/index.js` and only `src/` is shipped.

## Commands

Tests require live PostgreSQL **and** SQL Server databases — there is no mock layer. Each run
drops and recreates `test-json-schema-entity`, `...2`, and `...3` databases on both engines.

```sh
docker compose up -d                 # start postgres + mssql (+ a dev container)
POSTGRES_HOST=localhost POSTGRES_PASSWORD=postgres \
MSSQL_HOST=localhost MSSQL_PASSWORD=Passw0rd npm test
```

Or run the whole suite inside the dev container, where the host/password env vars are already
set (see `docker-compose.yml`):

```sh
npm run ssh                          # docker exec into the dev container's fish shell
npm test                             # inside the container
```

- Single test / subset: `npm test -- --grep "update cadastro"` (mocha `--grep` against the
  `describe`/`it` titles in `test/spec.js`). The suite runs every spec twice — once per dialect
  (`describe('postgres')` / `describe('mssql')`) — so a grep usually still hits both engines.
- The entry point `test/index.js` does global DB setup/teardown; always invoke tests through it
  (`mocha ... test/index.js`), not by pointing mocha at an individual spec file.
- Lint: `npx eslint .`. Style is enforced by Prettier (`.prettierrc`): no semicolons, single
  quotes, no bracket spacing, `printWidth` 70, arrow parens avoided.

CI (`.circleci/config.yml`) runs the same `npm test` against postgres + mssql service
containers, then auto-publishes to npm on `master` when `package.json`'s version differs from
the published one. Bump the version to release.

## Architecture

Everything of substance lives in `src/`. The `json-schema-table` and `sql-view` modules were
recently inlined from separate npm packages (`src/json-schema-table.js`, `src/sql-view.js`) and
are now local — do not reintroduce them as external dependencies.

### Layers

- **`src/index.js`** (the bulk of the code) — the entity engine. A module call
  `jse(name, schema, config)` builds an *entity definition* via the internal `entityFactory`.
  `.hasMany()` / `.hasOne()` attach child entity definitions recursively; every mutation
  (`setProperties`, `hasMany`, `foreignKey`, `useTimestamps`, …) calls `rebuild()` →
  `buildTable()` to recompute the cached derived data (`propertiesList`, primary keys,
  coercion functions, prebuilt INSERT/UPDATE/DELETE command templates, the `TableRecordSchema`).
- **`src/sql-view.js`** — turns a fetch criteria (`where`, `order`, paging) plus an
  adapter-built base query into a parameterized SELECT. Dialect-specific (`sqlView(dialect)`).
- **`src/json-schema-table.js`** (+ `json-schema-table-utils.js`) — DDL. Translates a JSON
  schema into `CREATE TABLE` / column + foreign-key `ALTER`s. Used by the entity's
  `createTables()` (create all tables in the tree) and `syncTables()` (add missing
  columns/refs). `createTables` then `syncTables` is the required order — refs are added after
  all tables exist.
- **`src/adapters/`** — per-dialect SQL generation behind a common interface. `common.js`
  holds the shared `create`/`update`/`destroy` execution logic and timestamp helpers;
  `postgres.js` and `mssql.js` provide `wrap` (identifier quoting: `"x"` vs `[x]`),
  command builders, type coercion, and `createTimestamps`. `getAdapter(dialect)` selects one;
  `'pg'`→`'postgres'`, `'ms'`→`'mssql'`, default `postgres`.
- **`src/entity-error.js`** — `EntityError`, a typed error (`type`, `message`, `errors[]`)
  thrown throughout for validation, missing records, invalid operations, etc.

### Runtime object model (`index.js`)

- An entity definition is a plain object of cached metadata (`data`) plus a `public` API and
  internal `methods`. `entityInstance = jse(...).new(db)` binds it to a `pg-cr-layer` /
  `mssql-cr-layer` connection and exposes `fetch`, `create`, `update`, `destroy`,
  `createInstance`, `createTables`, `syncTables`.
- **`TableRecord`** is the live document instance returned to callers. Its column values live in
  WeakMap-backed private state (`tableRecordData`/`tableRecordInfo`), exposed through
  getters/setters generated by **`TableRecordSchema`**, which also enforces enums, `date`
  normalization, and per-field `validator` rules on assignment. Each instance carries an
  immutable `.was` snapshot used to diff associations on update.
- `create`/`update`/`destroy` (module-level functions) recurse through `data.associations`,
  diffing `is` vs `was` by primary key to decide which child rows to create/update/delete, and
  wrap the whole operation in a DB transaction unless `options.transaction` is supplied.
- Lifecycle **hooks** (`beforeCreate`/`afterSave`/… and their `Delete`/`Destroy` aliases) and
  **validations** (`validate(id, fn, {onCreate,onUpdate,onDestroy})`) are registered on the
  definition and run via `runHooks`/`validateModel`/`validateFields`. Hook and validation fns
  may be generators yielding promises (run by the small `runGenerator` co-style helper).

### Schema property conventions

Property objects are JSON-schema-ish with extensions consumed across the layers: `field` (DB
column name when it differs from the attribute), `$ref` (foreign-key target, drives association
foreign keys), `autoIncrement`, `primaryKey`, `enum`, `maxLength`, `decimals`, `format`
(`'hidden'` excludes a column from the public schema), `defaultValue`, `timezone`, and
`mapper: {read, write}` (transform values on the way out of / into the DB). `useTimestamps()`
adds an `updated_at` column (suffix optional) surfaced as `updatedAt`, used for optimistic
concurrency — updates/deletes match on it and throw `RecordModifiedOrDeleted` on mismatch.

Test schemas in `test/schemas/` (many with accented/non-ASCII names, intentionally) are the
best reference for real-world property usage; `test/spec.js` exercises the full lifecycle.
